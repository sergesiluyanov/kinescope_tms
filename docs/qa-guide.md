# Гайд для QA: как вести тест-кейсы в TMS

Этот документ описывает, как устроена внутренняя TMS Kinescope с точки
зрения QA-инженера: какая модель данных, как писать кейсы и шаги, как
заводить баги, как запускать прогоны, как импортировать кейсы из xlsx,
и какие договорённости приняты в команде.

Если ты только что присоединился(ась) — пройди этот документ сверху
вниз, по ходу будут ссылки на конкретные экраны и API.

## 1. Иерархия объектов

```
Проект (Project)
└── Раздел (Section) — древовидно, любой уровень вложенности
    └── Тест-кейс (TestCase)
        └── Шаги (Steps): action + expected
```

И отдельно сбоку:

- **Test Run** — прогон выбранных кейсов; внутри лежат «снимки»
  (snapshot) кейсов на момент создания рана.
- **Bug** — баг-репорт, может быть привязан к конкретному кейсу.

### 1.1. Проекты

Один проект — это, как правило, один продукт или направление
(например, `Player`, `Dashboard`, `API`). Внутри проекта живёт всё:
разделы, кейсы, баги, прогоны.

Создавать проекты могут роли `admin` и `qa_lead`.

### 1.2. Разделы

Разделы — дерево. Используй их, чтобы группировать кейсы по фиче,
сценарию, экрану. Хорошие примеры:

```
Player
├── Воспроизведение
│   ├── HLS
│   ├── DASH
│   └── Live
├── Аналитика
└── Чат
```

Плохие примеры (стоит избегать):

- Разделы по приоритету (`P0`, `P1`, `P2`) — для этого есть поле
  «Приоритет» внутри кейса.
- Разделы по фамилии автора — для этого есть поле автора в системе.
- Разделы по тип-релизу (`smoke`, `регресс`) — для этого используются
  **теги** и **прогоны**, см. ниже.

Раздел можно **переименовать**, **перенести в другое место** или
**удалить вместе с потомками**. Переносить полезно, когда из плоской
структуры понадобилось сделать иерархию (или наоборот).

### 1.3. Тест-кейс

Главная единица. У каждого кейса:

- **Заголовок** (`title`, до 300 символов) — короткая формулировка,
  что именно проверяем.
- **Предусловия** (`preconditions`) — состояние системы и пользователя
  ДО первого шага.
- **Шаги** (`steps`) — упорядоченный список пар «действие → ожидаемый
  результат». Каждый шаг до 2000 символов в каждом поле.
- **Приоритет** (`priority`) — `low / medium / high / critical`.
- **Статус** (`status`) — `active` (по умолчанию) или `archived`.
- **Теги** (`tags`) — список произвольных меток.

У кейса автоматически проставляется автор, даты создания и обновления.

## 2. Как формулировать кейсы и шаги

### 2.1. Один кейс — одна проверка

Один кейс отвечает за одну проверяемую гипотезу.

Плохо:

> Заголовок: «Регистрация»
> Шаги: пройти регистрацию, залогиниться, сменить пароль, выйти,
> снова войти.

Хорошо: четыре отдельных кейса, у каждого свой заголовок и свои
шаги. Один из них — позитивный, остальные — негативные.

### 2.2. Заголовок описывает результат, а не действие

Сравни:

- Плохо: «Нажать на кнопку Регистрация».
- Хорошо: «При валидных данных пользователь регистрируется и
  получает письмо для активации».

Из заголовка должно быть понятно, **что именно мы проверяем** и
**какой результат ожидаем**. Это особенно важно для отчётов после
прогона — ты будешь видеть только заголовок.

### 2.3. Предусловия — это состояние, а не действия

Предусловия — это то, что должно быть **уже выполнено** до начала
теста. Не нужно туда писать «открыть форму регистрации» — это часть
шагов.

Хорошо:

- «Пользователь авторизован как `admin`».
- «В рабочей зоне есть проект с одним загруженным видео».
- «Браузер не имеет cookies от предыдущей сессии».

### 2.4. Шаги: одно действие → одно ожидание

Каждый шаг — это короткое действие (одна-две операции максимум) и
непосредственный результат этого действия. Если результат проверяется
только в самом конце сценария (общий итог) — поставь `expected`
только у последнего шага и оставь пустым у промежуточных.

Хорошо:

| Шаг | Действие                          | Ожидаемый результат                         |
| --- | --------------------------------- | -------------------------------------------- |
| 1   | Перейти на `app.kinescope.dev`    | Открывается главная                          |
| 2   | Нажать «Зарегистрироваться»       | Открывается форма регистрации с пустыми полями |
| 3   | Ввести валидную почту и пароль    | (пусто — промежуточный шаг)                  |
| 4   | Нажать «Зарегистрироваться»       | Появляется экран подтверждения, отправлено письмо |

### 2.5. Ожидаемый результат должен быть проверяемым

«Корректно работает», «всё хорошо», «никаких проблем» — это **не**
ожидаемый результат. Любой исполнитель прогона должен мочь однозначно
ответить «прошло / не прошло». Конкретика:

- Числа и пороги: «Количество видео уменьшилось ровно на 1».
- Конкретные строки UI: «На странице появляется тост “Видео удалено”».
- Коды и статусы: «API возвращает 200 OK и поле `id` в ответе».
- Состояние данных: «В таблице `videos` запись со статусом `deleted`».

### 2.6. Антипаттерны

- **Перекрёстные ссылки между кейсами**. «Выполнить шаги кейса
  TC-0042, затем продолжить здесь» — плохо. Если набор шагов
  повторяется в нескольких сценариях — это либо общие предусловия,
  либо стоит вынести в отдельную автотест-фикстуру. Каждый кейс
  должен быть самостоятельным.
- **Зависимость от состояния от предыдущего кейса**. Кейс должен
  проходить в любом порядке.
- **Шаги-«заметки»**: «не забыть проверить ...» — это не шаг, это
  чек-лист либо отдельный кейс.

## 3. Приоритеты и статусы

### 3.1. Приоритеты

| Значение  | Когда ставить                                                       |
| --------- | ------------------------------------------------------------------- |
| `critical`| Ломает бизнес-сценарий целиком: нельзя зарегистрироваться, нельзя загрузить видео, теряются данные, security. Должен проходить в каждом релизе. |
| `high`    | Важный пользовательский сценарий, без него релиз нельзя выкатить, но обходные пути есть. |
| `medium`  | Обычные функциональные сценарии, не критичные UX-флоу.              |
| `low`     | Косметика, edge-кейсы, очень редкие сценарии, проверки локализации. |

Если сомневаешься между двумя — поставь меньший приоритет. «Раздувание»
critical-кейсов делает приоритет бесполезным.

### 3.2. Статусы

- `active` — кейс актуален, включается в прогоны.
- `archived` — кейс больше не релевантен, но удалять жалко (история).
  Архивные кейсы не показываются в обычном списке прогонов, но их
  можно вернуть и переактивировать.

Не удаляй кейс, если он покрывал реальный сценарий, — лучше
**заархивируй**. Если сценарий полностью переехал в автотесты —
тоже архивируй с тегом `auto`.

## 4. Теги

Теги — это произвольные метки, ими удобно резать срезы. Договорённости
команды:

- `smoke` — входит в smoke-набор, прогоняется на каждой сборке.
- `regress` — входит в регресс на релиз.
- `auto` — покрыт автотестом (используется как ориентир для приоритезации).
- `flaky` — известная нестабильность, требует доработки.
- `pre-release` — обязательный для проверки перед публичным релизом.

Если хочется выделить функциональную область — используй теги типа
`upload`, `player`, `chat`, `dashboard`. Но не дублируй ими структуру
разделов: тег должен быть полезен **как фильтр поверх разделов**,
например «все smoke-кейсы из `Player` и `Dashboard`».

## 5. Прямые ссылки на кейсы

У каждого тест-кейса есть стабильный URL:

```
/projects/{projectId}/cases/{caseId}        # каноническая ссылка
/c/{caseId}                                 # короткая, для автотестов
```

Короткую ссылку использует автотест-фреймворк: в репорте упавшего
автотеста добавляется `https://<host>/c/12345`, и QA одним кликом
открывает соответствующий кейс. При открытии короткой ссылки TMS сама
резолвит, какому проекту и разделу принадлежит кейс, и редиректит на
канонический URL.

В UI рядом с каждым кейсом и в шапке формы редактирования есть кнопка
«Скопировать ссылку» (`🔗`).

Если нужно поделиться ссылкой в Slack/Telegram — копируй каноническую
ссылку, она читаемее.

## 6. Импорт тест-кейсов из XLSX

Если ты переезжаешь из Test IT или просто хочешь массово завести
кейсы из таблицы — используй кнопку **«⬆ Импорт XLSX»** на странице
`Тест-кейсы` любого проекта.

### 6.1. Формат файла

Поддерживается экспорт Test IT и «ручные» шаблоны в Excel/Google Sheets.
Парсер находит колонки **по заголовку**, поэтому порядок не важен, а
лишние колонки игнорируются. Каждая колонка распознаётся по нескольким
вариантам заголовков (см. правую часть таблицы):

| Поле                  | Что писать                                                                                             | Заголовки колонки (любой из)                                            |
| --------------------- | ------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------- |
| ID                    | Уникальный идентификатор кейса в файле. **Необязательно** — нужен только для многострочной формы (см. ниже). | `ID`                                                                    |
| Расположение          | Путь к разделу одной строкой через ` -> ` (или `/`). Несуществующие уровни создадутся.                  | `Расположение`, `Раздел`, `Path`, `Folder Path`                         |
| Раздел / Подраздел    | Альтернатива «Расположению»: верхний уровень и подраздел в **двух разных** колонках.                    | Верхний: `Section`, `Folder`, `Suite`. Под: `Subsection`, `Папка`, `Subfolder` |
| Название              | Заголовок кейса. Если пусто — соберётся из «ожидаемого результата» или будет `Кейс #<ID>`.              | `Наименование`, `Название`, `Заголовок`, `Title`                        |
| Предусловия           | Текст предусловий.                                                                                     | `Предусловия`, `Preconditions`, `Pre-conditions`                        |
| Шаги                  | **Обязательная колонка.** Несколько шагов в одной ячейке `1. ... 2. ...` разрезаются автоматически.    | `Шаги`, `Steps`                                                         |
| Ожидаемый результат   | Один на сценарий — навешивается на последний шаг; нумерованный список — попарно к шагам.                | `Ожидаемый результат`, `Expected`, `Expected Result`                    |
| Приоритет             | Русские названия и `P0/P1/P2/P3` (P0→critical, P1→high, P2→medium, P3→low). По умолчанию `medium`.     | `Приоритет`, `Priority`                                                 |
| Теги                  | Через запятую/`;`/перевод строки. Решётки `#` снимаются.                                               | `Тег`, `Теги`, `Tag`, `Tags`                                            |
| Заметки               | Произвольный текст — приклеится к предусловиям с префиксом «Заметки: …», чтобы информация не потерялась. | `Notes`, `Заметки`, `Комментарии`, `Comments`                          |

Поддерживаются **две формы записи** кейсов в файле:

1. **Один кейс — одна строка** (типично для ручных шаблонов). `ID` может
   быть как заполнен (`SH-01`, `LOGIN-42`), так и отсутствовать как
   колонка целиком — тогда каждая строка с непустыми «Шагами» считается
   новым кейсом.
2. **Многострочный Test IT-формат.** Первая строка кейса содержит
   непустой `ID`, следующие строки с пустым `ID` добавляют шаги,
   предусловия и теги к этому же кейсу. Новый `ID` — начало нового
   кейса.

### 6.2. Опции импорта

- **Отбросить первый уровень разделов** — полезно, если экспорт всегда
  начинается с одного корня (`Kinescope -> ...`).
- **Разбивать слитные шаги в одной ячейке** — включено по умолчанию.
  Если выключить, многострочный текст шага останется одним шагом.

После выбора файла открывается превью: видно, сколько кейсов
загружено, какие разделы будут созданы, образец первых 20 кейсов.
Импорт начинается только по кнопке «Импортировать» — превью ничего
не меняет в БД.

### 6.3. Что важно знать перед импортом

- Импорт **не дедуплицирует** кейсы. Если запустить тот же файл
  дважды — получите кейсы в двух экземплярах. Делать массовое
  «обновление» через xlsx сейчас нельзя — только добавление.
- Если в xlsx нет «Наименования» (как в большинстве экспортов Test IT),
  заголовок собирается автоматически: сначала из «общего ожидаемого
  результата», затем из первого шага, иначе — `Кейс #<ID из файла>`.
  Дубли заголовков допустимы (например, для группы API-кейсов
  «HTTP 200 OK»).
- Если файл большой (> 1000 кейсов) — рассчитывай на 10–30 секунд
  ожидания. Бэкенд комитит всё в одной транзакции, поэтому при ошибке
  посередине ничего лишнего в БД не остаётся.

## 7. Прогоны (Test Runs)

Прогон — это набор кейсов, который ты собираешь под конкретное
тестирование: smoke, регресс, hotfix-чек, проверка фичи и т.п.

### 7.1. Как устроен прогон

1. Создаёшь Run, выбираешь разделы (с подразделами или без), даёшь имя
   и описание, опционально — окружение (`stage`, `prod`).
2. На момент создания **все кейсы снэпшотятся в Run**: их title,
   preconditions, steps, теги и приоритет копируются внутрь Run-айтемов.
3. Дальше прогон живёт собственной жизнью: даже если кейс был
   изменён или удалён в основном дереве, **в текущем Run он остаётся
   как был**. Это нужно, чтобы метрики прогона не «плыли».
4. Каждый айтем прогоняешь со статусом `passed / failed / blocked / skipped`,
   добавляешь комментарий.
5. Из проваленного айтема можно одной кнопкой «Создать баг» — баг
   откроется с предзаполненной ссылкой на кейс.

### 7.2. Соглашения по статусам

- `passed` — кейс прошёл как ожидалось.
- `failed` — кейс не прошёл. **Обязательно** должен сопровождаться
  багом или ссылкой на уже существующий. Без бага `failed` — это
  пробел в процессе.
- `blocked` — нельзя выполнить из-за внешней причины (среда не работает,
  невозможно подготовить данные). Тоже желателен комментарий или
  ссылка на блокирующий баг.
- `skipped` — кейс намеренно пропущен (например, фича выключена за
  фича-флагом). С комментарием.

### 7.3. Когда создавать новый прогон

- На каждый релиз — отдельный прогон, чтобы хранить историю.
- На каждый hotfix — мини-прогон только из релевантных кейсов.
- На крупную фичу — прогон с её кейсами (можно по тегу).

### 7.4. HTML-отчёт по прогону

На странице прогона в шапке есть три кнопки:

- **⬇ HTML** — скачивает self-contained HTML-файл со сводкой:
  название, окружение, статус, автор, даты, длительность и progress-bar
  по статусам (passed / failed / blocked / skipped / untested). Открывается
  в любом браузере, можно распечатать (`@media print` заложен).

- **🔗 Поделиться / Копировать ссылку** — генерирует **публичную**
  share-ссылку и сразу копирует её в буфер. Ссылка выглядит как
  `https://tms.kinescope.dev/api/v1/public/test-runs/<token>/report.html`
  и открывается **без логина** — удобно кинуть менеджеру/продакту.
  Токен один на прогон и не меняется от повторных нажатий, поэтому
  ранее разосланная ссылка не «отваливается».

- **✕** (появляется, когда ссылка активна) — отзывает публичную ссылку.
  Прежний токен перестаёт работать; при следующем «Поделиться»
  сгенерируется новый.

**Отличия версий отчёта:**

- **⬇ HTML (скачать)** — «полный» отчёт: помимо шапки и сводки
  включает секцию **«Проблемные кейсы»** — карточку по каждому failed
  и blocked с приоритетом, исполнителем, датой прохождения, ссылкой
  на связанный баг (`BUG-####`) и **комментарием QA**. Пустой
  комментарий помечается как «Комментария нет» — так проще искать
  провалы, где не оставили пояснения. Если к кейсу привязан баг —
  под комментарием разворачивается его карточка с описанием, шагами
  воспроизведения, фактическим и ожидаемым результатом, окружением,
  severity и статусом. Отчёт можно распечатать (стили под `@media
  print`), и все детали останутся на месте.

- **🔗 Публичная share-ссылка** — «компактный» отчёт: только шапка и
  сводка со статистикой. Без списка кейсов и без комментариев.
  Это осознанное решение: публичная ссылка не должна проливать
  наружу конкретику по багам и внутренние заметки QA. Если нужно
  прислать кому-то детали — скачай `HTML` и отправь файлом.

## 8. Баг-репорты

Бэг привязан к проекту (опционально — к конкретному кейсу) и имеет:

- `severity` (`blocker / critical / major / minor / trivial`)
- `priority` (`high / medium / low`)
- `status` (`new / in_progress / fixed / wont_fix / verified / closed`)
- `environment`, `steps_to_reproduce`, `actual_result`,
  `expected_result`, `tags`, `assignee`.
- Ссылку на Kaiten card (когда подключена интеграция).

### 8.1. Минимально достаточный баг

В баге должно быть достаточно информации, чтобы разработчик мог его
воспроизвести и пофиксить **без дополнительных вопросов**. Минимум:

1. Заголовок: «Что сломано, в каких условиях».
2. Окружение: версия, браузер/устройство, окружение (`stage`, `prod`).
3. Шаги воспроизведения: кратко, по делу.
4. Фактический результат.
5. Ожидаемый результат.

И прикреплённый кейс, если баг найден в рамках конкретной проверки.

### 8.2. Severity vs Priority

- **Severity** отражает «насколько серьёзный сам дефект» — техническая
  тяжесть.
- **Priority** — «как срочно его чинить». Например, security-уязвимость
  с низкой вероятностью эксплуатации может иметь `critical severity`,
  но `medium priority`, потому что бизнес-импакт прямо сейчас нулевой.

Не выставляй `blocker severity` без согласования. Это исключительные
случаи, когда без фикса нельзя релизиться вообще.

## 9. Чек-листы

Чек-лист — это плоский список коротких пунктов «проверить X», без
шагов и предусловий. В TMS отдельной сущности «чек-лист» нет, но мы
их представляем как **обычные кейсы с одним шагом**:

```
title: "Все 6 базовых метрик отображаются"
steps:
  - action:    "Проверить: на экране видны Video bitrate, Audio bitrate, FPS, Packet loss, Streamer IP, RTMP server"
    expected:  "Все шесть полей видны и содержат осмысленные значения"
priority: high
tag:      "checklist,smoke"
```

Чек-листы удобно складывать в отдельный подраздел `Чек-листы` внутри
основного раздела фичи, чтобы быстро запускать прогон только по
чек-листу.

## 11. Рабочий процесс: типовые сценарии

### 11.1. «Появилась новая фича, нужно её покрыть»

1. Зайди в проект, который относится к этой фиче.
2. Создай раздел (или подраздел в существующем) с понятным именем.
3. Заведи кейсы по сценариям: happy-path, основные ветвления,
   негативы, edge-case (по одному на проверку).
4. Поставь приоритеты и теги.
5. При желании сразу собери прогон по этой фиче и прогони его до
   релиза — это твой acceptance.

### 11.2. «Поломался автотест, нужно понять, что покрывал кейс»

В отчёте автотеста есть ссылка `/c/{caseId}`. Открываешь, видишь
ожидание, шаги, прикреплённые баги. Если поломка повторяется —
заводишь баг прямо из проваленного айтема в текущем прогоне.

### 11.3. «Появился баг с прода — нужно покрыть кейсом»

1. Заводишь баг в проекте, привязываешь к существующему кейсу
   (если такой есть) или создаёшь новый кейс с воспроизведением.
2. Тегаешь кейс `regression`, чтобы он попадал в регресс на каждом
   релизе.
3. После фикса меняешь статус бага на `fixed`, прогоняешь кейс,
   ставишь айтему статус `passed`, бэг переводишь в `verified`.

### 11.4. «Нужно отдать набор кейсов внешней команде»

Сейчас экспорт из TMS планируется, но ещё не реализован. Временный
обходной путь — поделиться ссылками на конкретные кейсы из UI или
запросить через API:

```
GET /api/v1/sections/{section_id}/test-cases
GET /api/v1/test-cases/{case_id}
```

С Bearer-токеном из своей сессии.

## 12. Что НЕ нужно делать

- Не дублируй один и тот же кейс в нескольких разделах. Используй
  теги или перенос раздела.
- Не пиши в шагах «убедитесь, что всё в порядке» — это не
  проверяемый результат.
- Не оставляй баги без ссылок на кейсы. Раз нашёл баг — есть либо
  существующий кейс, который его поймал, либо нужен новый.
- Не используй приоритет `critical` как «надо обратить внимание».
  `critical` — это блокирование релиза.
- Не удаляй раздел, в котором ещё лежат живые кейсы, ради «чистки».
  Сначала перенеси кейсы в новый раздел, потом удаляй.
- Не правь чужие кейсы без согласования владельца раздела/фичи.
  Лучше — создай свой и обсуди с автором.
